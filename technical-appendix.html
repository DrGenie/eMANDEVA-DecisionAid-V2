Overview

1. Purpose and scope

This appendix documents the inputs, assumptions, calculations, and user-facing outputs implemented in eMANDEVAL-Future. The tool supports structured exploration of potential future COVID-19 vaccine mandate designs in Australia, France, and Italy by combining discrete choice evidence (predicted acceptability) with user-supplied epidemiological and costing assumptions.

The tool provides:

  • Model-based predicted public support for a mandate configuration (mandate versus opt-out).
  • Expected lives saved in the covered population from a user input expressed per 100,000 people.
  • Monetised benefits using a configurable single “benefit metric” (for example VSL, VSLY, value per QALY, or health system savings).
  • Total costs, net benefit, and benefit–cost ratio (BCR), conditional on user-entered costs.
  • Lives-saved equivalents (MRS) translating preference weights for design features into “lives saved per 100,000” units.
  • Scenario storage, comparison, export, and briefing/AI prompt generation for policy workflows.

eMANDEVAL-Future is a transparent decision aid for exploring trade-offs. It is not a forecasting engine and does not replace legal advice, ethical deliberation, distributional analysis, or operational feasibility assessment.

Workflow

2. Workflow and tabs (including policy modes)

2.1. Policy modes (question-first workflow)

The tool supports a question-first workflow intended for policy meetings. Users select a policy question mode such as:

  • Compare a strict versus lenient mandate.
  • Maximise support subject to BCR ≥ 1.
  • Maximise BCR subject to support ≥ X%.
  • Compare country A versus B under similar designs.

Policy modes do not change the underlying equations or coefficients. They guide the user by suggesting scenario templates and highlighting which parameters are most relevant to the chosen objective.

2.2. Settings

Settings apply to the current configuration and to saved scenarios (a scenario stores the settings active at the time of saving):

  • Analysis horizon: number of years over which costs and benefits are accumulated.
  • Population covered: size of the population exposed to the mandate.
  • Currency label: display label for monetary outputs (for example AUD, EUR).
  • Benefit metric: scalar monetary value applied to the implied health impact.

2.3. Mandate configuration

The configuration defines the discrete choice model frame and the policy attributes:

  • Country (Australia, France, Italy).
  • Outbreak scenario (mild/endemic versus severe outbreak).
  • Scope (high-risk occupations only versus all occupations and public spaces).
  • Exemption policy (medical only; medical + religious; medical + religious + personal belief).
  • Coverage threshold to lift mandate (50%, 70%, 90% vaccinated).
  • Expected lives saved per 100,000 (user input).

2.4. Costing

Costs are entered as totals for the selected analysis horizon (or equivalently as inputs that the user interprets as horizon totals). The tool supports country- and outbreak-specific default cost suggestions, scaled by population and horizon, which users can load and edit.

2.5. Results, saved scenarios, and briefings

Results combine the above inputs to produce lives saved, monetised benefits, total cost, net benefit, BCR, and predicted support, plus interpretive elements: status chips, charts, and lives-saved equivalents (MRS).

Saved scenarios enable comparison and export. Pinned scenarios (up to three) appear in a comparison dashboard for rapid side-by-side viewing.

A “presentation mode” is included for workshops. It enlarges key text and charts and reduces non-essential visual clutter, while preserving all calculations and stored scenarios.
Interpretation

3. Predicted public support (mixed logit simulation)

3.1. Model output definition

The tool reports “predicted public support” as the model-based probability that an average respondent would choose the configured mandate over the opt-out (no mandate) option, conditional on the selected country and outbreak scenario.

3.2. Utility specification used in the web tool

For each simulated preference draw β, utilities are computed as:

U(mandate; β) = ASCPolicyA + βscopeAll·1(scope=all) + βexMedRel·1(exempt=med+rel) + βexMedRelPers·1(exempt=med+rel+pers) + βcov70·1(cov=70%) + βcov90·1(cov=90%) + βlives·(lives saved per 100k)

U(opt-out; β) = ASCOptOut

P(mandate | β) = 1 / (1 + exp(-(U(mandate; β) − U(opt-out; β))))

3.3. Simulation and reproducibility

The mixed logit implies random coefficients. The tool simulates predicted support by drawing coefficients from independent normal distributions using the stored mean and standard deviation for each coefficient and frame. For R draws:

Predicted support = (1 / R) · Σr=1…R P(mandate | βr).

The current implementation uses R = 1000 draws and a fixed pseudo-random seed so that results are reproducible within a build. This means that the predicted support is deterministic given the same configuration, coefficients, and seed.

Interpretation: a predicted support of 65% indicates that, under the stated frame and assuming the DCE generalises, about two thirds of respondents would choose the mandate over opt-out. It is not a forecast of compliance, enforcement success, or political feasibility.

Methods

4. DCE, mixed logit specification, and coefficients

4.1. Attributes and levels represented in the tool

eMANDEVAL-Future represents mandate options using:

  • Scope (high-risk occupations only vs all occupations and public spaces).
  • Exemption policy (medical only; medical + religious; medical + religious + personal belief).
  • Coverage threshold to lift the mandate (50%, 70%, 90% vaccinated).
  • Expected lives saved per 100,000 (continuous attribute).

4.2. Mixed logit implementation in the web tool

The web implementation uses the mixed logit mean and standard deviation parameters for each country and outbreak frame. Random coefficients are generated as:

βk,r = μk + σk · zk,r, where zk,r ~ Normal(0,1) and k indexes coefficients.

The implementation uses independent z draws per coefficient and draw. Correlation across coefficients is not represented in the current web tool.

Costs & benefits

5. Cost taxonomy, benefit metrics, and derived outcomes

5.1. Cost categories

Costs are entered under a standard taxonomy intended to map to regulatory impact and implementation budgeting:

  • Digital systems and infrastructure
  • Communications and public information
  • Enforcement and compliance
  • Adverse-event monitoring and compensation
  • Administration and programme management
  • Other mandate-specific costs

Default cost suggestions are stylised and expressed as per-million-population annual values by country, adjusted by an outbreak multiplier, then scaled by:

Cost scale factor = (population / 1,000,000) × (analysis horizon in years) × (outbreak multiplier).

Default costs are included to help users start scenario construction. They are not a substitute for local costing and should be calibrated against jurisdiction-specific budget data, programme accounts, or regulatory impact statements before formal use.

5.2. Benefit metrics

The tool monetises the health impact using a single scalar “benefit metric” selected in Settings. Options include:

  • VSL: value per statistical life saved.
  • VSLY: value per statistical life-year gained.
  • Value per QALY: monetary value per quality-adjusted life-year gained.
  • Health system savings: an average monetary saving per averted death (or per equivalent health outcome), if a user chooses to interpret it that way.

In the current core implementation, the health impact input is “lives saved per 100,000 people”. The tool converts this to total lives saved:

Lives saved (total) = (lives saved per 100,000) × (population / 100,000).

Monetised benefits are computed as:

Benefit (monetary) = Lives saved (total) × (benefit metric value).

5.3. Extended outcome metrics (optional)

Version 2.1 supports optional reporting of additional outcomes such as hospitalisations averted, ICU admissions averted, and working days preserved. These outcomes can be entered directly per 100,000 or derived via user-specified conversion assumptions (for example, “1 life saved corresponds to X hospitalisations averted”).

The tool does not enforce “correct” conversion values because these are context-specific and depend on epidemiological conditions, population structure, variant, baseline risk, and healthcare capacity. If conversion factors are used in a formal setting, they should be justified and cited in the accompanying documentation.

5.4. Derived economic outputs

Given total cost and monetised benefit, the tool computes:

  • Net benefit: benefit − cost.
  • BCR: benefit / cost (reported as “not defined” when cost is zero or missing).
Lives-saved equivalents

6. Lives-saved equivalents (MRS)

Lives-saved equivalents translate preference weights for mandate design features into the common unit of “lives saved per 100,000”. This supports plain-language interpretation of how strongly respondents value a design change relative to the lives-saved attribute.

MRSx,LS = − βx / βLS.

The tool computes MRS using the mean coefficients for the selected country and outbreak frame, for:

  • Scope shift (high-risk only → all occupations and public spaces).
  • Exemption policy changes (medical only → medical + religious; medical only → medical + religious + personal belief).
  • Coverage threshold changes (50% → 70%; 50% → 90%).
Interpretation rule used in narratives: positive MRS indicates a design change that is less preferred (equivalent to “losing” lives saved); negative MRS indicates a design change that is more preferred (equivalent to “gaining” lives saved).
Scenario comparison

7. Saved scenarios, pinned dashboard, deltas, and exports

7.1. What is stored in a scenario

Each saved scenario stores a complete snapshot of inputs and outputs:

  • Settings (population, horizon, currency label, benefit metric type, benefit metric value).
  • Configuration (country, outbreak frame, scope, exemptions, coverage, lives saved per 100,000).
  • Costs (by category, if entered).
  • Derived outputs (total lives saved, benefit, cost total, net benefit, BCR, predicted support).
  • Optional distributional notes (Section 8).

Scenarios are stored in the browser using local storage. They persist across page refreshes in the same browser profile unless the user clears them.

7.2. Pinned scenario comparison dashboard

Users may “pin” up to three scenarios to a comparison dashboard for rapid side-by-side review. The dashboard includes:

  • A compact label per pinned scenario.
  • A radar-style visual comparing: predicted support, BCR, lives saved, and total cost.
  • Traffic-light stripes reflecting the same thresholds used in status chips (Section 9.2).

7.3. Deltas

When comparing pinned scenarios, the tool displays numerical deltas that highlight trade-offs (for example changes in support in percentage points, changes in BCR, changes in lives saved, and changes in cost). These deltas are descriptive and are not decision rules.

7.4. Exports and briefing outputs

The tool supports:

  • CSV export (Excel-readable) containing one row per scenario.
  • Word briefing export with structured scenario summaries and a short interpretation paragraph per scenario.
  • Auto-generated briefing text for the current scenario (quick copy/paste into emails or notes).
  • AI prompts for single-scenario briefs or comparative memos when multiple scenarios exist.

Exported text is intended as a starting point. Users should incorporate feasibility, legal and ethical considerations, and any distributional assessment.

Equity

8. Distributional notes and equity flagging

8.1. Purpose

Quantitative outputs alone can obscure distributional consequences. eMANDEVAL-Future includes structured fields that allow users to record distributional considerations alongside numerical results.

8.2. Fields captured

Groups most affected

Examples: specific occupations, age groups, migrants, low-income households, those with limited access to primary care.

Sectors most exposed

Examples: healthcare, education, hospitality, transport, social care, public service delivery.

Equity concerns (flag)

If this field is populated, the scenario is flagged to remind decision-makers that distributional impacts warrant explicit assessment.

8.3. How the equity flag is used

If an equity concern is recorded, the tool:

  • Flags the scenario in the saved scenarios table (visual indicator).
  • Includes distributional notes in Word exports and AI prompts (where applicable).
  • Encourages explicit discussion of mitigation options (for example targeted exemptions, access support, or compensatory measures).

The equity flag is a documentation aid. It does not calculate distributional welfare, inequality indices, or subgroup-specific utilities in the current web build.

Visual outputs

9. Visual outputs and thresholds

9.1. Cost–benefit chart

The tool shows a simple bar chart for benefit, cost, and net benefit, using the current currency label. Values are computed from Section 5.4.

9.2. Support chart (stacked mandate vs opt-out) and thresholds

Predicted support is visualised as mandate share versus opt-out share. Thresholds are indicated for interpretive support:

Support < 50%: low acceptability Support 50–70%: moderate acceptability Support ≥ 70%: high acceptability

These thresholds are descriptive labels intended for rapid communication. They should not be treated as formal decision criteria without additional justification.

9.3. BCR interpretation thresholds

BCR < 0.8: unfavourable BCR 0.8–1.0: uncertain BCR ≥ 1.0: favourable

9.4. MRS chart

Lives-saved equivalents are visualised as horizontal bars (positive values: less preferred changes; negative values: more preferred changes), based on Section 6.

9.5. Presentation mode

Presentation mode is designed for live workshops. It increases readability, emphasises the comparison dashboard and results panels, and maintains identical computation logic.

Reproducibility

10. Reproducibility, storage, and privacy

10.1. Reproducibility of predicted support

Predicted support uses a fixed random seed and a deterministic pseudo-random number generator. Within a given build, this ensures:

  • The same configuration yields the same predicted support value.
  • Scenario results can be reproduced when exported and reloaded (assuming the build and coefficients are unchanged).

10.2. Local storage

Saved scenarios are stored locally in the browser. This means:

  • Scenarios are not uploaded by the tool (unless the page is embedded within an environment that adds its own telemetry, which is outside the tool logic).
  • Clearing browser storage or using a different device/browser profile will remove or separate saved scenarios.
  • Exports (CSV/Word) produce files that can be shared, archived, or appended to policy documentation.

10.3. Practical data governance

The tool is intended for aggregate scenario planning. Users should avoid entering personally identifying information into free-text fields (including distributional notes), unless they have an explicit governance reason to do so.

Worked examples

11. Worked examples (illustrative)

These examples illustrate the workflow. They are not recommendations and should be replaced by jurisdiction-appropriate inputs and assumptions.

11.1. Italy, severe outbreak

  • Settings: population = 5,000,000; benefit metric = VSL; value per unit = EUR 5,000,000; horizon = 1 year.
  • Configuration: outbreak = severe; scope = all; exemptions = medical + religious; coverage = 70%; lives saved = 25 per 100,000.
  • Costs (illustrative): enter totals across the six cost categories for the horizon.

Apply configuration, load default costs (optional), edit costs, then review Results and save the scenario. Pin it for comparison if you are building a set of alternatives.

11.2. Australia, mild outbreak (frame sensitivity)

Hold the mandate design fixed and change only the outbreak scenario from mild to severe. Compare the predicted support and MRS outputs. This isolates how framing (via country- and outbreak-specific coefficients) shifts predicted acceptability.

11.3. Cross-country comparison with pinned dashboard

Construct aligned scenarios for Australia, France, and Italy (same scope, exemptions, coverage, and lives saved per 100,000). Save each, pin all three, and use the radar and deltas to communicate how predicted support and BCR differ across settings under comparable designs.

Governance

12. Versioning, quality control, and limitations

12.1. Versioning principles

The tool uses explicit versioning and a build date. This appendix is updated alongside releases so that exported scenario briefs can cite the build used to generate numbers.

12.2. Internal quality checks

Before release, a set of reference scenarios should be run both in the web tool and in the original estimation environment (for example R or MATLAB) to confirm:

  • Predicted support matches to expected rounding differences given the same seed and draw count.
  • Lives saved and monetised benefits match simple scaling calculations.
  • BCR and net benefit calculations match hand-checked examples.
  • MRS calculations match −β/βLS using the displayed coefficient means.

12.3. Limitations and responsibilities

  • DCE evidence is stated preference and may not fully reflect future behaviour, compliance, or political dynamics.
  • The lives-saved input is exogenous to the tool; it must be supplied from epidemiological modelling or empirical estimates.
  • Cost defaults are stylised and require calibration before formal use.
  • Distributional notes are qualitative documentation aids, not formal equity-weighted evaluation.

Responsibility for selecting inputs, documenting assumptions, and interpreting outputs rests with the analyst or institution using the tool.
